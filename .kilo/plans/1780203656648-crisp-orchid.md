# Workflow Analysis: Admin Upload for Customer Download

## Current Implementation Analysis

### 1. Are artwork_files_json files customer-owned uploads or admin-owned uploads?

**Both.** The `artwork_files_json` field stores image files that can be uploaded by either:
- **Customer**: via `app/my-orders/[orderId]/page.js` lines 413-500 - customer uploads artwork images
- **Admin**: via `app/admin/orders/[id]/page.js` lines 107-137 - admin uploads artwork images

Both use the same API endpoint `/api/order-items/[id]/artwork` (route.ts lines 13-22) which accepts uploads from either role.

### 2. Where is the file stored when admin uploads artwork from `app/admin/orders/[id]/page.js`?

Files are stored in `uploads/private-artwork/order-{order_id}/` directory (route.ts lines 80-81):
- Temp upload goes to `uploads/private-artwork-temp/` via `/api/artwork/temp-upload` (temp-upload/route.ts line 10)
- Then moved to `uploads/private-artwork/order-{order_id}/{timestamp}-{filename}` (artwork/route.ts line 90)

### 3. Does the uploaded file become visible to the customer automatically?

**Yes.** The file becomes visible to the customer immediately because:
- The `artwork_files_json` JSON array is updated in the database
- Both admin and customer pages read from the same source
- The customer page `/api/order-items/[id]/artwork/[index]` (artwork/[index]/route.ts lines 32-37) allows both admin and customer access

### 4. Can admin upload a completely new file that was not originally uploaded by the customer?

**Yes.** The admin upload form in `app/admin/orders/[id]/page.js` (line 570-589) allows uploading files without any requirement for prior customer upload. The `replaceArtwork: true` flag (line 122) replaces existing artwork with new content.

### 5. Are there existing database fields intended for proof files, admin attachments, customer downloadable files, production files?

**No dedicated fields exist for these specific purposes.** Current database fields (from `migration_order_item_artwork.sql` and `migration_order_item_requirements.sql`):

- `artwork_files_json` - for artwork images (both customer and admin can upload)
- `custom_size_note` - text notes
- `requirement_files_json` - for customer-uploaded requirements documents
- `requirement_status` - approval workflow
- `requirement_uploaded_at`, `requirement_reviewed_at` - timestamps

No separate fields for proof files, admin attachments, or production files.

### 6. Is there any existing API that allows admin upload and customer download of the SAME file?

**Yes.** The same files are shared via:
- **Upload**: `/api/order-items/[id]/artwork` (PATCH) - allows both admin and customer (artwork/route.ts lines 18-22)
- **Download**: `/api/order-items/[id]/artwork/[index]` (GET) - allows both admin and customer (artwork/[index]/route.ts lines 32-52)

### 7. Show the exact customer page/component where admin-uploaded files become visible

`app/my-orders/[orderId]/page.js`:
- Lines 453-483: Artwork files are displayed and downloadable
- Lines 457-470: Uses `/api/order-items/${item.id}/artwork/${i}` for viewing and `?download=1` for download

Also `app/my-orders/page.js`:
- Lines 379-407: Lists artwork files with download links using same API pattern

### 8. Does Requirement #3 already exist?

**Yes, Requirement #3 exists.** The workflow in the code allows:
- Admin uploads artwork → stored in `private-artwork/order-{id}/` → becomes immediately visible to customer on "My Orders" page
- Customer can download via `/api/order-items/{id}/artwork/{index}?download=1`

However, there's a **critical limitation**: The current implementation has no distinction between customer-uploaded artwork and admin-uploaded proofs. Both are stored in the same `artwork_files_json` field and displayed with identical UI labels ("Artwork files", "Download artwork").
